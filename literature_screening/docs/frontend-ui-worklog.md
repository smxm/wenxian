# 前端交互界面实施过程文档

## 目标

在不破坏现有模块边界的前提下，提供一个可直接交互的前端界面，使用户能够完成以下任务：

1. 导入一批文献文件与筛选标准
2. 运行初筛流程
3. 查看初筛统计与输出文件
4. 基于初筛结果生成独立报告模块的简洁报告
5. 在一个界面中完成主要工作，而不是手动拼接命令

## 边界约束

- 主项目 `src/literature_screening` 继续只负责初筛
- 独立报告模块 `separated_modules/formal_report_module` 继续单独维护
- 前端只做编排与展示，不直接复制两边的业务逻辑
- 后续如果初筛或报告模块参数变化，优先通过服务层适配，而不是让 UI 直接依赖底层实现细节

## 方案选择

当前计划采用 `Streamlit` 作为本地交互前端。

原因：

- 本地启动成本低，适合当前单机工作流
- 支持文件上传、文本输入、状态展示和 Markdown 预览
- 便于快速搭出一个可用且可持续修改的界面
- 后续可通过独立服务层与现有模块解耦

## 解耦策略

准备新增一层前端应用服务层，负责：

- 将 UI 输入转换为稳定的运行请求对象
- 生成临时配置和输出目录
- 调用初筛模块
- 调用独立报告模块
- 统一读取输出摘要、报告和文献文件

前端界面不直接操作底层 prompt、parser、pipeline 细节。
