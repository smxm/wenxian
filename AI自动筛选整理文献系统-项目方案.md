# AI自动筛选整理文献系统项目方案

## 1. 项目概述

### 1.1 项目名称
AI自动筛选整理文献系统

### 1.2 项目背景
用户在进行系统综述、专题调研或研究报告撰写时，通常会从不同数据库、不同检索词组下导出多批次 BibTeX 文献。现有人工处理流程普遍存在以下问题：

- 多批次文献来源分散，合并成本高
- 重复文献较多，人工去重低效且不稳定
- 标题与摘要筛选耗时长
- 筛选结论难以标准化记录
- 最终纳入文献、排除文献与未使用文献难以同步整理

本项目拟开发一个自动化脚本模块，作为完整文献筛选整理系统的一部分，用于将用户提供的多个 BibTeX 文件统一整合、去重、分批提交给 AI 模型执行标准化筛选，并输出结构化报告与 BibTeX 结果文件。

### 1.3 项目目标
构建一个可复用、可追踪、可扩展的文献筛选脚本，实现以下能力：

- 接收多个 BibTeX 文件作为输入
- 合并并去重所有文献记录
- 按固定批次大小分批提交给 AI 模型筛选
- 根据用户定义的纳入/排除标准对文献进行判断
- 在达到用户指定的纳入数量目标后自动停止后续筛选
- 输出纳入文献报告、排除文献报告、运行总结和 BibTeX 结果文件
- 保留未被使用到的剩余文献，便于后续继续筛选

---

## 2. 适用场景

本模块适用于以下典型场景：

- 系统综述或范围综述前期文献初筛
- 多关键词、多数据库检索结果汇总筛选
- 研究报告撰写前的相关文献候选池整理
- 需要限定纳入数量上限的快速证据收集
- 需要保留筛选过程痕迹与依据的学术工作流

---

## 3. 项目范围

### 3.1 本阶段纳入范围
本阶段重点实现一个命令行脚本级别的最小可行系统，覆盖以下内容：

- BibTeX 文件读取
- 多文件合并
- 基础去重
- 文献信息标准化
- 分批筛选
- Kimi API 调用
- AI 筛选结果解析
- 达标停止逻辑
- 报告与 BibTeX 导出
- 日志与中间结果落盘

### 3.2 本阶段暂不纳入范围
以下功能不作为当前阶段必须实现内容：

- Web 前端或桌面界面
- 数据库存储
- 高级模糊去重策略
- 全自动多轮复筛
- 人工复核工作台
- 多模型统一调度框架
- 与 Zotero、EndNote 等软件直接集成

## 4. 核心业务需求

### 4.1 输入需求
系统应支持以下输入：

- 一个或多个 `.bib` 文件
- 用户筛选标准
- 每批提交的文献数量，默认 20
- 目标纳入数量，例如纳入满 50 篇后停止
- 模型配置参数，测试阶段默认使用 Kimi API
- 可选配置：
  - 是否启用严格去重
  - 是否保留不确定结果
  - 是否保存原始模型响应
  - 输出目录位置

### 4.2 处理需求
系统应支持以下处理流程：

- 自动读取多个 BibTeX 文件
- 统一解析为结构化文献对象
- 对重复文献执行合并
- 将文献池按批次切分
- 构建标准化提示词并发送给 AI
- 解析 AI 返回结果
- 更新文献状态
- 达到目标纳入数后停止处理
- 标记剩余未处理文献

### 4.3 输出需求
系统至少输出以下结果：

- 纳入文献报告
- 排除文献报告
- 运行总结报告
- 纳入文献 BibTeX 文件
- 剩余未使用文献 BibTeX 文件
- 中间处理日志和批次原始结果

---

## 5. 总体流程设计

整体流程建议如下：

1. 接收输入文件与运行配置
2. 解析多个 BibTeX 文件
3. 合并文献并执行去重
4. 构建统一文献池
5. 按每批 20 篇切分文献
6. 调用 AI 进行批量筛选
7. 解析返回结果并更新状态
8. 判断是否达到目标纳入数量
9. 若达到目标则停止，否则继续下一批
10. 汇总纳入、排除、不确定与未使用文献
11. 生成报告与导出 BibTeX 文件

---

## 6. 业务流程分解

### 6.1 阶段一：输入接收
用户输入可以来自单次运行命令、配置文件或后续上层系统。

建议支持的输入项：

- `input_files`
- `screening_criteria`
- `batch_size`
- `target_include_count`
- `model_provider`
- `model_name`
- `output_dir`
- `save_raw_response`
- `allow_uncertain`

### 6.2 阶段二：BibTeX 合并
系统读取多个 `.bib` 文件后，将所有条目统一转为内部结构化记录。处理时需要保留来源信息，便于后续溯源。

每条记录建议保留：

- 原始 bibtex 文本
- 原始 bib key
- 来源文件名
- 解析后的标准字段

### 6.3 阶段三：文献去重
去重策略建议分层进行，以平衡稳定性和复杂度。

第一层为强去重：

- DOI 完全一致
- PMID 完全一致
- arXiv ID 完全一致

第二层为基础弱去重：

- 标题标准化后完全一致
- 标题高度相似且年份一致
- 标题高度相似且第一作者一致

当前阶段建议先实现强去重和标题标准化完全匹配，后续版本再加入相似度算法。

### 6.4 阶段四：文献标准化
为了让 AI 更稳定地判断，每篇文献应被转换为统一的文献卡片格式。

建议保留字段：

- 标题
- 作者
- 年份
- 期刊或会议名称
- DOI
- 摘要
- 关键词
- 来源信息

如摘要缺失，应显式标记为空，而不是忽略该字段。

### 6.5 阶段五：分批筛选
系统从未处理文献池中按顺序抽取文献，默认每批 20 篇，组装成一个 AI 请求。

每批应具备以下信息：

- `batch_id`
- 该批文献列表
- 使用的筛选标准
- 使用的模型参数
- 请求时间

### 6.6 阶段六：AI 决策解析
AI 返回结果应使用结构化格式，建议强制为 JSON。每篇文献至少包含：

- `paper_id`
- `decision`
- `reason`
- `evidence`
- `confidence`

其中 `decision` 推荐限定为：

- `include`
- `exclude`
- `uncertain`

### 6.7 阶段七：停止条件判断
每完成一批后，应统计当前已纳入数量。

如果已纳入数量大于或等于用户设定目标，则：

- 停止后续 AI 调用
- 将尚未处理的文献全部标记为 `unused`
- 输出最终报告和结果文件

说明：最后一批处理后允许纳入数量略微超过目标值，不强求精确卡在目标数上。

### 6.8 阶段八：结果导出
系统最终应导出：

- 纳入文献报告
- 排除文献报告
- 运行总结
- 纳入文献 BibTeX
- 剩余未使用文献 BibTeX
- 可选的不确定文献报告

## 8. 去重策略设计

### 8.1 去重目标
将来自不同检索词、不同数据库导出的同一篇文献合并为单一记录，同时保留来源痕迹。

### 8.2 当前阶段建议策略

优先级 1：

- DOI 相同直接判定为同一篇

优先级 2：

- 标题标准化后完全一致

标题标准化建议包括：

- 转小写
- 去除多余空格
- 去除常见标点差异
- 去除 BibTeX 转义噪音

### 8.3 合并规则建议
若判定为同一篇，合并时应遵循：

- 优先保留非空摘要
- 优先保留更完整的作者字段
- 优先保留 DOI
- 累积 `source_files`
- 累积 `source_keys`
- 保留原始 BibTeX 备份

### 8.4 风险与后续扩展
后续可考虑加入：

- 基于标题相似度的模糊去重
- 作者名归一化
- 年份偏差容忍
- 期刊名标准化

---

## 9. AI筛选策略设计

### 9.1 筛选原则
AI 在本系统中承担“标题与摘要级别的初筛判断”角色，而不是最终人工审稿替代者。因此系统设计应强调：

- 结构化输出
- 理由可追踪
- 标准一致性
- 支持不确定项

### 9.2 单篇文献输出要求
每篇文献建议要求 AI 输出：

- 是否纳入
- 简要判断理由
- 判断依据字段
- 置信度

### 9.3 输出格式要求
必须优先采用 JSON 输出，避免自由文本导致后处理困难。

推荐返回结构示意：

```json
{
  "batch_id": "batch_001",
  "results": [
    {
      "paper_id": "paper_0001",
      "decision": "include",
      "reason": "研究对象与筛选主题直接相关，摘要显示包含目标机制分析。",
      "evidence": [
        "title",
        "abstract",
        "keywords"
      ],
      "confidence": 0.89
    }
  ]
}
```

### 9.4 不确定结果的必要性
建议保留 `uncertain` 状态，原因如下：

- 有些摘要信息不足，不适合强制二选一
- 可降低错误排除高价值文献的风险
- 方便后续人工复核

---

## 10. 停止逻辑设计

### 10.1 目标停止机制
用户可以设置目标纳入数量，例如：

- 纳入满 30 篇后停止
- 纳入满 50 篇后停止

系统应在每批筛选完成后判断是否满足停止条件。

### 10.2 剩余文献处理
若达到目标后停止：

- 不再继续调用模型
- 未处理文献标记为 `unused`
- 导出剩余未使用文献的 BibTeX 文件

### 10.3 统计口径
需要明确区分三类文献：

- 已处理且纳入
- 已处理且排除
- 尚未处理且未使用

这三类在统计和报告中必须分开。

## 11. 输出物设计

### 11.1 报告输出
建议至少输出以下文件：

- `included_report.md` 或 `included_report.json`
- `excluded_report.md` 或 `excluded_report.json`
- `run_summary.json`
- `uncertain_report.json`（可选）

### 11.2 BibTeX 输出
建议输出以下文件：

- `included.bib`
- `unused_remaining.bib`
- `excluded.bib`（可选）

### 11.3 日志与中间结果
建议保存以下过程文件：

- `merged_records.json`
- `deduped_records.json`
- `batch_001_input.json`
- `batch_001_output.json`
- `raw_api_response_batch_001.txt`

这样便于调试、审计和复现。

## 12. 技术方案建议

### 12.1 推荐语言
推荐优先使用 Python。

原因包括：

- 处理 BibTeX 和文本数据较方便
- 数据清洗、结构化处理成本低
- 调用 AI API 简洁
- 报告导出灵活
- 后续扩展测试与自动化较顺手

### 12.2 推荐模块划分

```text
project/
  data/
    input/
    output/
    cache/
    logs/
  src/
    bibtex/
      parser.py
      deduper.py
      exporter.py
    screening/
      prompt_builder.py
      kimi_client.py
      batcher.py
      result_parser.py
    reporting/
      report_generator.py
      summary.py
    core/
      models.py
      config.py
      pipeline.py
    main.py
  tests/
  README.md
```

### 12.3 API模型选择
测试阶段使用 Kimi API 即可，原因是：

- 便于快速验证流程
- 成本与调用复杂度相对可控
- 适合作为提示词和结果解析链路的验证模型

后续可扩展支持其他模型提供方。

## 15. 风险分析

### 15.1 BibTeX 数据质量风险
风险表现：

- 字段缺失
- DOI 缺失
- 标题编码异常
- 作者字段格式混乱

应对建议：

- 做字段标准化
- 明确保留原始数据
- 允许空字段但要显式标记

### 15.2 AI 输出稳定性风险
风险表现：

- JSON 不合法
- 漏判或漏字段
- 批次间标准漂移

应对建议：

- 强制结构化输出
- 结果解析校验
- 失败自动重试
- 保存原始响应

### 15.3 筛选标准歧义风险
风险表现：

- 用户标准过于模糊
- 模型理解前后不一致

应对建议：

- 将筛选标准结构化
- prompt 中明确纳入与排除条件
- 保留不确定状态

### 15.4 提前停止带来的统计偏差
风险表现：

- 剩余文献未处理但被误当作排除

应对建议：

- 明确 `unused` 状态
- 在报告中单列剩余文献

## 17. 验收标准

本阶段系统若满足以下条件，可视为达成目标：

- 能成功读取多个 BibTeX 文件
- 能完成基础去重并输出去重后结果
- 能按固定批次调用 Kimi API
- 能正确解析结构化筛选结果
- 能在达到目标纳入数量后停止
- 能正确区分纳入、排除、不确定与未使用文献
- 能导出纳入报告、排除报告、运行总结和 BibTeX 文件
- 出错时有足够日志支持排查

## 18. 结论

本项目适合采用“先打通主链路，再逐步增强”的实现路线。优先完成 BibTeX 合并去重、批量 AI 筛选、达标停止和结果导出这四个核心能力，可以快速形成一个可用的最小系统。待流程稳定后，再迭代加入模糊去重、人工复核、多模型支持和可视化界面等增强功能。

该方案兼顾了研究流程中的可追踪性、可解释性与工程实现难度，适合作为后续完整 AI 文献筛选整理系统的基础模块。
